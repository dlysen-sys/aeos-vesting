THE GUARD (Why This Happens)
================================================================================

safeApprove() Logic:

    function safeApprove(IERC20 token, address spender, uint256 value) {
        require(
            (value == 0) || (token.allowance(...) == 0),
            "SafeERC20: approve from non-zero to non-zero allowance"
        );
        // ... do the approve
    }

This blocks:  X → Y  (where X ≠ 0 and Y ≠ 0)
This allows:  0 → Y  (always OK)
This allows:  X → 0  (always OK)

Why? Prevent ERC-20 approve race condition:

    OLD BEHAVIOR (unsafe):
    User: "Reduce allowance from 100 to 50"
    approve(spender, 50)

    Attacker: "Increase allowance to 200"
    approve(spender, 200)

    If Tx2 mines first:
    1. Spender spends 100 (front-run)
    2. Tx1 mines: allowance = 50
    3. Tx2 mines: allowance = 200
    4. Spender gets 100 + 200 = 300 🚨

SOLUTION: Block 100 → 50 → 200 in same account


THREE FUNCTIONS EXPLAINED
================================================================================

┌─ approve(spender, amount) ────────────────────────────────────────────────┐
│ Raw ERC-20. No checks.                                                    │
│ Works: 1st call ✅, 2nd call ✅, 3rd call ✅                             │
│ Risk: Front-run vulnerable                                               │
│ Use: Only when you know what you're doing                                │
└─────────────────────────────────────────────────────────────────────────┘

┌─ safeApprove(spender, amount) ────────────────────────────────────────────┐
│ DEPRECATED. Has the guard.                                               │
│ Works: 1st call ✅, 2nd call ❌, 3rd call ❌                             │
│ Risk: Blocks legitimate repeated approval patterns (← THIS ISSUE)         │
│ Use: Never. OpenZeppelin marks as deprecated                             │
└─────────────────────────────────────────────────────────────────────────┘

┌─ safeIncreaseAllowance(spender, amt) ──────────────────────────────────────┐
│ Smart: Only adds delta, never overwrites.                                 │
│ Works: 1st call ✅, 2nd call ✅, 3rd call ✅                             │
│ Safe: No race condition possible                                          │
│ Use: When you want to add to existing allowance                          │
└─────────────────────────────────────────────────────────────────────────┘

┌─ forceApprove(spender, amount) ───────────────────────────────────────────┐
│ Smart: Resets to 0 first, then approves.                                 │
│ Works: 1st call ✅, 2nd call ✅, 3rd call ✅                             │
│ Safe: No race condition. Always works.                                   │
│ Use: Modern best practice (OZ v4.9.3+)                                   │
│ Requires: @openzeppelin/contracts@^4.9.3                                 │
└─────────────────────────────────────────────────────────────────────────┘


THE FIX (Choose One)
================================================================================

OPTION A: forceApprove() [RECOMMENDED] ─────────────────────────────────────
  Change:    usdtToken.safeApprove(address(liquidity), usdtForLiquidity);
  To:        usdtToken.forceApprove(address(liquidity), usdtForLiquidity);
  Lines:     1
  Compat:    OZ v4.9.3+
  Result:    ✅ Works forever
  Requires:  npm install @openzeppelin/contracts@latest


OPTION B: Manual Reset [COMPATIBLE] ───────────────────────────────────────
  Change:    usdtToken.safeApprove(address(liquidity), usdtForLiquidity);
  To:        usdtToken.safeApprove(address(liquidity), 0);
             usdtToken.safeApprove(address(liquidity), usdtForLiquidity);
  Lines:     2
  Compat:    All versions
  Result:    ✅ Works forever
  Requires:  No version check


OPTION C: safeIncreaseAllowance() [EXPLICIT] ──────────────────────────────
  Change:    usdtToken.safeApprove(address(liquidity), usdtForLiquidity);
  To:        uint256 current = usdtToken.allowance(address(this), address(liquidity));
             if (current > 0) usdtToken.safeDecreaseAllowance(address(liquidity), current);
             usdtToken.safeIncreaseAllowance(address(liquidity), usdtForLiquidity);
  Lines:     4
  Compat:    All versions
  Result:    ✅ Works forever, most explicit
  Requires:  No version check


EXECUTION FLOW: Before vs After
================================================================================

BEFORE (safeApprove) ──────────────────────────────────────────────────────

User 1 Purchase:
  ├─ allowance = 0
  ├─ safeApprove(500)
  │  └─ Check: (500 == 0) || (0 == 0)? TRUE ✅
  │  └─ approve(500) → allowance = 500
  └─ Success ✅

User 2 Purchase:
  ├─ allowance = 200 (residual from User 1)
  ├─ safeApprove(250)
  │  └─ Check: (250 == 0) || (200 == 0)? FALSE ❌
  │  └─ REVERT
  └─ Failure ❌


AFTER (forceApprove) ──────────────────────────────────────────────────────

User 1 Purchase:
  ├─ allowance = 0
  ├─ forceApprove(500)
  │  ├─ approve(500) → allowance = 500 ✅
  └─ Success ✅

User 2 Purchase:
  ├─ allowance = 200 (residual)
  ├─ forceApprove(250)
  │  ├─ Try: approve(250)
  │  ├─ If fails: approve(0) → approve(250)
  │  └─ allowance = 250 ✅
  └─ Success ✅

User 3 Purchase:
  ├─ allowance = 150 (residual)
  ├─ forceApprove(350)
  │  └─ allowance = 350 ✅
  └─ Success ✅
